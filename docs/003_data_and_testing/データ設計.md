# データ設計

## 基本方針

在留資格申請のデータは、次の3層に分ける。

1. `case_data`: 案件の正本。AI/OCR/ヒアリング/人手補正/レビュー結果を保存する。
2. `form_definitions`: 入管オンライン申請フォームの項目台帳。画面上の `field_id`、`field_name`、選択肢、必須、条件付き項目を保持する。
3. `application_data`: Chrome拡張へ渡す投入用JSON。`case_data` と `form_definitions` / `mappings` から生成する派生物。

`application_data.json` を案件管理の正本にしない。これはブラウザ入力用の薄いデータであり、実務判断、OCR根拠、レビュー状態、添付資料管理を保持する場所ではない。

テスト用途では、`fixtures/` のバッチケースから `case_data` を正規化する段階と、`fixtures_single/` の単票ケースから `application_data` を生成する段階を分ける。前者は複数申請人を含む案件構造の検証、後者は1申請人=1フォームの投入検証に使う。

## 正規 case_data

`case_data` は、申請画面の項目順ではなく、案件・実務判断の単位で持つ。

主なトップレベル構造。

- `case`: 案件ID、流入経路、申請種別、対象在留資格、ワークフロー状態。
- `application`: 入国目的、予定日、希望期間、活動内容詳細。
- `applicant`: 申請人の基本情報。
- `passport`: 旅券情報。
- `residence_card`: 在留カード情報。更新・変更案件で重要。
- `immigration_history`: 出入国歴、COE申請歴、犯罪歴、退去強制歴。
- `family`: 同伴者、在日親族、同居者。
- `education`: 学歴。
- `transcript_subjects`: 成績科目。技人国の職務整合性チェックで使う。
- `employment_history`: 職歴。
- `qualifications`: 語学・資格。
- `employer`: 所属機関・雇用会社。
- `supporting_documents`: 添付資料の受領、OCR、再依頼状態。
- `assessments`: AIまたは人の判断結果。

定義は [case_data.schema.json](/Users/yohei/Documents/1.起業/31.%20在留資格申請/visa_application_app/data/schemas/case_data.schema.json) に置く。デモ正本は [demo_case_data.json](/Users/yohei/Documents/1.起業/31.%20在留資格申請/visa_application_app/data/cases/demo_case_data.json)。

## フォーム台帳

[rasens_offer_fields.json](/Users/yohei/Documents/1.起業/31.%20在留資格申請/visa_application_app/data/form_definitions/rasens_offer_fields.json) がフォーム項目台帳の正本。

この台帳は `reference_form.html` から抽出したもので、次を保持する。

- 項目番号
- ラベル
- `field_id`
- `field_name`
- `input_type`
- `select` の選択肢
- 必須項目
- 条件付き項目

17.2〜17.4 のような条件付き項目、21.2〜21.8 のような在日親族の繰り返し項目、職歴01〜06のような繰り返し項目もフォーム台帳側で保持する。

## マッピング

[rasens_offer_mapping.json](/Users/yohei/Documents/1.起業/31.%20在留資格申請/visa_application_app/data/mappings/rasens_offer_mapping.json) は、`case_data` の安定キーをフォーム項目へ対応させる。

例。

```json
{
  "canonical_id": "immigration_history.latest_entry.start_date",
  "value_path": "immigration_history.latest_entry.start_date",
  "form_item_no": "17.3",
  "field_name": "item[31].textData",
  "field_id": "switch_256127",
  "input_type": "text",
  "transform": "date_yyyymmdd"
}
```

`field_id` や `field_name` は画面依存なので、案件正本のキーには使わない。画面変更時は `form_definitions` と `mappings` を更新し、`case_data` は維持する。

条件付き項目は `visible_when` で表す。例えば17.2〜17.4は、17.1「過去の出入国歴 有無」が有の場合だけ投入対象になる。

繰り返し項目は `groups` で管理する。例えば在日親族21.2〜21.8は最大6枠、職歴26.1〜26.6も最大6枠の物理フォーム項目として存在するが、`case_data` 側では配列として扱う。

現時点のマッピングは主要項目の初期版。フォーム全項目の正は `form_definitions/rasens_offer_fields.json` にあり、マッピングは必要に応じて増やす。

## デモJSON

デモ用には2種類のJSONを置く。

- [demo_case_data.json](/Users/yohei/Documents/1.起業/31.%20在留資格申請/visa_application_app/data/cases/demo_case_data.json): 架空案件の正本。
- [demo_application_data.json](/Users/yohei/Documents/1.起業/31.%20在留資格申請/visa_application_app/data/generated/demo_application_data.json): Chrome拡張に渡す投入用JSON。

`demo_application_data.json` は手編集せず、次のコマンドで生成する。

```bash
python3 visa_application_app/scripts/build_application_data.py \
  visa_application_app/data/cases/demo_case_data.json \
  visa_application_app/data/mappings/rasens_offer_mapping.json \
  visa_application_app/data/generated/demo_application_data.json
```

## 運用上の注意

- 実案件データとデモデータを混ぜない。
- `fixtures/` は抽出・正規化用、`fixtures_single/` はフォーム投入用と役割を分ける。
- 個人情報を含む実案件の `case_data` は、共有範囲と保存場所を明確にする。
- 公開・共有可能なfixtureは、合成データまたは匿名化済みデータに限定する。
- 実案件raw資料は制限付きデータとして扱い、必要に応じてリポジトリ外の保管領域へ分離する。
- `unknown`、空欄、`not_applicable` は区別する。未回収と非該当が混ざると不足確認が壊れる。
- `confidence=medium` やOCR根拠が弱い項目は人確認に回す。
- Chrome拡張の `application_data.json` は同梱デモ・検証用。案件正本ではない。
- 最終送信は自動化しない。入力後は人が確認してから手動で進める。
