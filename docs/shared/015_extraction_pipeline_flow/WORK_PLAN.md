# スコープ分割型抽出 作業計画

## 背景と目的

現在の抽出パイプラインは Gemini API を1回呼び出しで全フィールドを抽出している。
以下の課題がある：
- response_schema の状態数制約で配列（職歴×6社、在日親族×6名）が追加困難
- 申請フォーム274項目中、現在46項目しか抽出していない（カバー率約25%）
- フォーム構造と抽出スキーマの対応が不明確

本計画では、RASENSフォームのセクション構造に合わせてGemini呼び出しを分割し、段階的にカバレッジを拡大する。

## スコープ定義

### Gemini抽出スコープ（6スコープ）

| ID | スコープ名 | RASENSフォーム | フィールド数 | 状態数 | Phase |
|---|---|---|---|---|---|
| S1 | 身分事項 | No.1〜20 | ~30 | ~64 | P1 |
| S2 | 所属機関＋雇用条件＋活動内容 | No.2〜11 | ~35 | ~70 | P1 |
| S3 | 学歴＋資格 | No.23〜25 | ~8 | ~18 | P1 |
| S4 | 在日親族（配列、0〜2名） | No.21 | ~15 | ~29 | P2 |
| S5 | 職歴（配列、0〜3社） | No.26 | ~22 | ~43 | P2 |
| S6 | レビュー | — | ~5 | ~20 | P1 |

### 固定データ（Gemini不要）

| 項目 | ソース | 備考 |
|---|---|---|
| 取次者（5項目） | アカウント設定（下記参照） | 全案件共通。初回設定後は自動投入 |
| 代理人（6項目） | 企業マスター or 案件入力 | 企業の担当者 |
| 高度専門職ポイント表（57項目） | 全項目0で自動投入 | 技人国では不使用 |
| 受領方法（3項目） | 固定値「郵送」 | 変更頻度極低 |
| 入力確認（4項目） | UI操作 | チェックボックス＋顔写真 |

### スコープ外

- **整合性レビューの自動ループ**: 抽出結果の品質チェック→不足フィールドの再抽出→再チェック...というフィードバックループ。将来的にはレビュー（S6）の結果を見て、source_refsカバレッジが低いスコープを自動再実行する仕組みを入れたいが、設計・実装ともに大きいため今回はスコープ外とする。現段階では「1回抽出→レビュー→人が確認」のフローで進める。
- **LangGraph 導入**: 上記フィードバックループの実装基盤として将来検討。現段階では asyncio.gather による並列実行で十分。
- **入力確認画面（/confirm）**: RASENSへの投入前の最終確認画面。スコープ分割とは独立した機能のため別計画。
- **Gemini Context Caching**: 現状の書類量（12K-15Kトークン）では最低ライン32Kに届かないため見送り。

## 実装計画

### Phase 1: コア抽出の分割（S1, S2, S3, S6）

目的: 現在の1回呼び出しを4+1回に分割し、並列実行で同等以上のレイテンシを維持しつつ、フィールドカバレッジを拡大。

#### Step 1-1: スコープ別スキーマ定義
- 担当: Engineer
- ファイル: `visa-app/backend/extractors/schema.py`
- 内容: SCOPE1_IDENTITY_SCHEMA〜SCOPE6_REVIEW_SCHEMA を定義。既存の EXTRACTION_SCHEMA は後方互換で残す
- 受け入れ条件: 各スキーマの状態数がGemini制約内に収まること

#### Step 1-2: スコープ別プロンプト
- 担当: Engineer
- ファイル: `visa-app/backend/extractors/prompt_template.py`
- 内容: スコープごとの抽出指示テンプレートを追加。書類フィルタ定義（SCOPE_DOCUMENT_ROLES）
- 受け入れ条件: 各スコープのプロンプトが対象セクションのフィールドのみを抽出指示すること

#### Step 1-3: 並列実行オーケストレーション
- 担当: Lead Engineer
- ファイル: `visa-app/backend/extractors/gemini.py`, `visa-app/backend/main.py`
- 内容:
  - `extract_scoped()` 関数追加（async）
  - `extract_all_scopes()` 関数追加（asyncio.gatherで S1,S2,S3を並列 → S6を逐次）
  - `_start_gemini_extraction()` に `scoped=True` パラメータ追加
  - 結果マージロジック
- 受け入れ条件: 並列実行でウォールクロック時間が現行と同等以下（25秒以内）

#### Step 1-4: 互換レイヤー確認
- 担当: QA
- 内容: フロントエンド変更なしで動作すること。case_data/field_metadata/reviewの構造が既存と同一であること
- テスト方法: 再抽出→ブラウザでレビュー画面確認→ハイライト動作確認

## 将来計画（スコープ外の詳細）

### 整合性レビュー自動ループ（将来）

現在のレビュー（S6）は1回実行して結果を人が確認するだけ。将来的には：

1. S6 の結果で source_refs カバレッジが閾値（90%）以下のスコープを検知
2. そのスコープだけを自動再実行（プロンプトに「前回の不足フィールド」を明示）
3. 再実行結果をマージして再度 S6 でチェック
4. 最大2回まで繰り返し、それでも不足なら人に委ねる

これは asyncio + カスタムロジックで40-50行程度で実装可能だが、品質チェックの閾値設計やリトライ戦略の検討が必要。Phase 2 完了後に着手を検討する。

### LangGraph 移行（将来）

上記ループが複雑化した場合（条件分岐5つ以上、Human-in-the-Loop が必要）、LangGraph に移行して宣言的に管理する。現段階では asyncio.gather + カスタムロジックで十分。

## 技術リスク（Lead Engineer レビュー結果）

| リスク | 深刻度 | 対策 |
|---|---|---|
| **Firestore 部分更新の競合** | 高 | スコープ並行実行時、`case_data` を丸ごと update すると後勝ちで消える。Firestore のドット記法（`case_data.applicant`）でセクション単位更新するか、全スコープ完了後に一括保存する |
| **同期→async 中途半端化** | 高 | FastAPI の `def` エンドポイント内で `await` は使えない。`asyncio.to_thread()` で Gemini 呼び出しをラップするのが最もリスクが低い。Firestore/GCS の同期呼び出しはそのまま |
| **google-genai SDK の async** | 中 | `client.aio.models.generate_content()` が使えるかバージョン依存。使えなければ `asyncio.to_thread` でフォールバック |
| **既存テスト書き換え** | 中 | async 版関数を別名で追加し、同期版は残す移行戦略。既存テストを壊さない |

### 推奨: Phase 1 の実装方針

1. **エンドポイントは `def` のまま**（async 化しない）
2. **Gemini 並列呼び出しは `asyncio.to_thread` + `asyncio.gather`** で、同期関数を並列化
3. **Firestore 保存は全スコープ完了後に一括更新**（部分更新の競合を回避）
4. **既存の同期テストはそのまま維持**。新しいスコープ別テストを追加
