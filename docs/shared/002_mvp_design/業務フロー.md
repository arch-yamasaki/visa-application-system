# 業務フロー

## 1. 相談・依頼の入口

最初の連絡はメール、電話、LINE、紹介、CCでの巻き込みなど。営業で広く集めるというより、知り合い、既存顧客、人材紹介会社、事業協同組合、士業紹介から来る。

入口別の特徴。

- 外国人本人: 知り合いや紹介が中心。件数は少なめ。
- 雇用会社: 既存顧客、顧客の知人会社、顧問先など。今後スケールさせやすい。
- 人材紹介会社: フォースバレーのような紹介会社。申請人情報がまとまりやすい。
- 事業協同組合: 技能実習後に特定技能や技人国へ変えたい、過去に働いた人を呼びたいなど。
- 士業紹介: 弁護士や他士業が扱いきれない案件の相談。

## 2. 在留資格の見立て

まず、何をしたい案件かを確認する。

- 新規で外国人材を呼びたい
- 更新したい
- 変更したい
- 家族を呼びたい
- 配偶者系で申請したい
- 技能実習から特定技能・技人国へ移りたい

初期MVPでは、ここで技人国に該当しそうな案件を主対象にする。高度専門職、技能、興行、配偶者、家族滞在などは検知だけして、人の判断に回す。

## 3. 必要情報の収集

技人国で基本的に集める情報。

- 申請人の基本情報
- 旅券・パスポート
- 在留カード
- 履歴書・職務経歴書
- 卒業証明書
- 成績証明書
- 資格証明書、語学証明
- 雇用条件通知書、雇用契約書
- 雇用会社の基本情報
- 会社カテゴリー、決算・規模に関する資料
- 仕事内容、配属部署、活動内容
- 来日予定日、入社予定日、契約期間
- 配偶者・同伴家族・在日親族の有無

フォースバレー経由では申請人情報がExcelなどで来ることがある。一般案件ではフォームを用意し、足りない項目だけを再入力してもらう形が良い。

収集した情報は、申請画面の項目順ではなく、案件単位の `case_data` として正規化する。`case_data` には、申請人、旅券、在留カード、出入国歴、家族、学歴、成績科目、職歴、雇用会社、活動内容、添付資料、レビュー状態を保存する。

## 4. 不足・矛盾チェック

AIで最初にやるべきチェック。

- 必須資料が揃っているか
- 履歴書と卒業証明書の学校名が一致するか
- 履歴書と成績証明書の専攻・科目が仕事内容と合うか
- 雇用条件通知書の職務内容が技人国に合うか
- 在留希望期間と雇用契約期間が矛盾しないか
- ローマ字氏名、生年月日、国籍、旅券番号に転記ミスがないか
- 配偶者や同伴家族の扱いに漏れがないか
- 過去の来日歴、在日親族、退去強制歴など重要項目が未確認でないか

不足があれば、AIが追加依頼文を作る。回答先はメール、LINE、フォームのいずれでもよいが、最終的には `case_data` に構造化して保存する。

不足確認では、空欄、不明、非該当を分ける。例えば「在日親族なし」と「未確認」は別状態として扱う。

## 5. 技人国の実質判断

技人国では、単に書類が揃っているだけでは足りない。重要なのは、申請人が学んだこと・経験したことと、日本で行う仕事がつながっていること。

見るべき観点。

- 専攻・成績表に、業務に関係する科目があるか
- 業務内容が専門的・技術的・国際的な内容になっているか
- 現場作業、単純労働、調理などに見えないか
- 会社側の仕事内容説明が不利な言い方になっていないか
- 過去に通った同種案件の活動内容詳細を使えるか

ここはAIが一次チェックし、リスクありなら人が見る。

## 6. 申請書作成・オンライン入力

情報が揃ったら、オンライン申請システムに入力する。多くの項目は機械的に転記できる。

入力用データは `case_data` を直接手編集して作らない。フォーム台帳 `form_definitions` とマッピング `mappings` から `application_data` を生成し、Chrome拡張に渡す。

AI・自動化で担える部分。

- `case_data` からフォーム項目への自動マッピング
- ブラウザ入力補助
- 添付PDFの整理
- 会社情報の再利用
- 過去案件からの活動内容詳細の再利用
- 最終確認用の申請内容プレビュー
- 条件付き項目、繰り返し項目の投入対象判定

人が最終確認すべき部分。

- 在留資格の選択
- 活動内容詳細
- 理由書・補足説明
- リスク案件の申請方針
- 添付資料の過不足

## 7. 申請後

審査にはおおむね数か月かかる。COEの有効期間は許可後3か月という話があり、入社予定日・来日予定日から逆算して申請時期を決める必要がある。

補正・追加資料依頼が来た場合は、過去の入力、添付資料、判断履歴を参照して対応する。ここもAIが資料検索と回答案作成を支援できる。

申請後の対応履歴も `case_data` または案件配下のレビュー/補正履歴として残す。生成済み `application_data` は再生成できる派生物なので、補正の正本にはしない。
