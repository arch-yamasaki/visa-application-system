# AIエージェント要件

## MVP方針

初期MVPは技人国に絞る。全在留資格を一気に扱うより、件数が多く、型が比較的あり、フォースバレー経由などでデータも集まりやすい技人国の実務支援から作る。

MVPのゴールは、完全自動申請ではなく「担当者が判断しやすい状態まで案件を整える」こと。

データ設計上は、案件の正本を `case_data` とし、Chrome拡張に渡す `application_data` は生成物として扱う。OCR、ヒアリング、人手補正、レビュー結果、添付資料状態は `case_data` に集約し、画面入力直前に `mappings` から投入用JSONを生成する。

## 想定ユーザー

- 行政書士、補助者、事務スタッフ
- 外国人材を雇用する会社
- 人材紹介会社
- 事業協同組合

初期の操作主体は行政書士事務所側。外部の会社や紹介会社には、必要情報の入力フォームや追加依頼フォームを渡す。

## コア機能


### AIエージェント自体に必要な機能

### 1. 案件取り込み

- 添付資料、フォーム回答を案件単位で取り込む
- 新規、更新、変更、家族帯同などの種別を仮分類する
- 取り込んだ情報を `case_data` へ正規化する
- 未回収、非該当、不明を区別して保存する

### 2. 書類OCR・項目抽出

対象資料。

- パスポート
- 在留カード
- 履歴書
- 職務経歴書
- 卒業証明書
- 成績証明書
- 雇用条件通知書
- 雇用契約書
- 会社資料

抽出する項目。

- 氏名、ローマ字、国籍、生年月日、性別
- 旅券番号、有効期限
- 在留カード番号、在留期限
- 学校名、専攻、卒業日
- 成績表の科目
- 職歴、職務内容
- 雇用会社、勤務地、給与、契約期間
- 入社予定日、来日予定日
- 家族・配偶者・同伴者・在日親族

抽出結果は、フォームの `field_id` や `field_name` ではなく、`applicant.birth_date`、`applicant.passport.number`、`applicant.education[0].school_name` のような安定キーへ入れる。画面依存のキーは `form_definitions` と `mappings` 側だけで扱う。

### 3. 不足確認

・新規、更新、変更、家族帯同などの種別や申請職種の情報から、書類・項目の不足を自動で検出する。足りない場合は、相手に送る依頼文を作る。

例。
- 卒業証明書がありません
- 成績証明書に業務と関連する科目が見当たりません
- 雇用条件通知書に職務内容が不足しています
- 履歴書の学校名と卒業証明書の学校名が異なります
- 契約期間が1年なのに在留期間5年で申請するため、更新予定の補足が必要です

### 4. 技人国チェック

技人国として通りそうかを一次判定する。

チェック観点。

- 業務内容が技術・人文知識・国際業務の範囲か
- 学歴・専攻・成績科目と業務がつながるか
- 職歴と業務がつながるか
- 単純労働、現場作業、調理などに見えないか
- 翻訳・通訳の場合に語学力の説明があるか
- 活動内容詳細の文面が審査上不利でないか

出力は「OK」「要確認」「人の判断必須」の3段階がよい。

### 他機能

#### 5. 申請入力支援

- `case_data` から入力項目を事前生成
- オンライン申請システムへのブラウザ入力補助
- 添付ファイルの整理
- 申請前チェックリスト生成
- 申請確認PDF・控えの保存

完全自動送信は初期では避け、最終送信前に人が確認する。

入力支援の流れ。

1. `case_data` を正本として保存する。
2. `data/form_definitions/rasens_offer_fields.json` でフォーム項目を管理する。
3. `data/mappings/rasens_offer_mapping_v2.json` で正規キーとフォーム項目を対応させる。
4. visa-app backend の `/cases/{case_id}/application-data` が投入用 `rows` を作る。
5. Chrome拡張は backend から取得した `rows` を読み、画面へ投入する。

17.2〜17.4 のような条件付き項目は `visible_when`、21.2〜21.8 や職歴01〜06のような繰り返し項目は `groups` で扱う。初期実装では主要項目から始め、フォーム台帳を正としてマッピングを広げる。

#### 6. ナレッジ蓄積

過去案件情報を取得して、それをAIエージェントの機能とする。Chrome拡張だけに閉じ込めず、`case_data` とレビュー結果を蓄積する。バッチで実装すれば良いかも。

- 通った活動内容詳細
- 補正対応の履歴
- 不許可・リスク案件
- 会社ごとの申請履歴
- 職種ごとの必要資料
- 学歴・専攻・職務の対応パターン

このナレッジが、AIの判断精度と事務所固有の強みになる。

## 人のレビューが必要な箇所

- 在留資格分類が技人国で確定できない案件
- 技人国と技能の境界案件
- 興行、配偶者、家族滞在、高度専門職、特定技能
- 学歴と職務が弱く、理由書が必要な案件
- 会社の決算・規模・カテゴリーに不安がある案件
- 中国人配偶者系など、運用が厳しくなっている案件
- 過去不許可・補正あり案件
