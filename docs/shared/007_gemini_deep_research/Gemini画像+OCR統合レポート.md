## 0. トークン計算の基礎知識

### 画像トークン（モデル世代による違い）

| モデル世代 | 条件 | トークン数/枚 |
|---|---|---|
| **2.5系** (Flash/Pro) | 全画像 | **固定 256 tokens**（解像度無関係） |
| **2.5系** HIGH設定時 | Pan & Scan 適用 | 最大 ~2,048 tokens |
| **3系** (3 Flash等) | `media_resolution=LOW` | 280 tokens |
| **3系** | `media_resolution=MEDIUM`（公式推奨） | 560 tokens |
| **3系** | `media_resolution=HIGH`（デフォルト） | 1,120 tokens |
| **3系** | `media_resolution=ULTRA_HIGH` | 2,240 tokens |

### PDFトークン

| モデル世代 | トークン数/ページ | 備考 |
|---|---|---|
| **2.5系** | **258 tokens 固定** | 画像として処理 |
| **3系** | 280〜1,120（`media_resolution`依存） | + ネイティブテキストは**追加コストなし**で抽出 |

> **重要**: Gemini 3系では、テキスト埋め込みPDF（ネイティブPDF）の場合、画像トークンとは別にテキストが自動抽出され、追加トークンコストがかからない。OCR不要でテキスト精度が高い。

## 1. OCRテキストのみ vs 画像のみ vs 両方併用 — トレードオフ比較

### 1.1 方式概要

| パターン | 方式 | 入力構成 |
|---|---|---|
| **A: テキストのみ** | Cloud Vision OCR → テキストを Gemini に渡す | プロンプト + OCRテキスト |
| **B: PDF直接** | PDF をそのまま Gemini に渡す（推奨） | プロンプト + PDF |
| **B': 画像変換** | PDF→画像変換して Gemini に渡す | プロンプト + 画像 |
| **C: テキスト+PDF** | OCRテキスト + PDF 両方渡す（精度最大化） | プロンプト + OCRテキスト + PDF |
| **C': テキスト+画像** | OCRテキスト + 画像変換 両方渡す | プロンプト + OCRテキスト + 画像 |

### 1.2 トレードオフ比較表

| 観点 | A: テキストのみ | B: PDF直接 | C: テキスト+PDF併用 |
|------|----------------|-----------|-------------------|
| **精度（テキスト書類）** | ◎ 高い（OCRが正確） | ○ 良好 | ◎ 最高（相互補完） |
| **精度（画像書類）** | △ レイアウト喪失 | ◎ 視覚情報保持 | ◎ 最高 |
| **座標・レイアウト取得** | ✗ 不可 | ○ 視覚的に判読 | ○ 視覚的に判読 |
| **コスト（2.5 Flash/100件）** | $2.63 | **$1.34** | $2.69 |
| **コスト（2.5 Pro/100件）** | $7.19 | **$5.37** | $7.43 |
| **入力トークン/申請人** | 8,500 | **2,935** | 10,435 |
| **実装難易度** | 中（OCRパイプライン要） | **低（PDF直送）** | 高（OCR+PDF両方） |
| **前処理の依存** | Cloud Vision API 必須 | なし | Cloud Vision API 必須 |
| **手書き・スタンプ対応** | ✗ OCR精度に依存 | ◎ 視覚で判読 | ◎ 視覚で判読 |

### 1.3 書類タイプ別の最適方式

| 書類タイプ | 特徴 | 最適方式 | 理由 |
|---|---|---|---|
| **パスポート** | 写真・MRZ・スタンプ | B: PDF直接 or C | 視覚情報が重要、MRZはOCRでも可 |
| **在留カード** | 写真・ホログラム | B: PDF直接 | 視覚情報必須 |
| **履歴書** | テキスト中心、手書き混在 | C: テキスト+PDF | 手書き部分はPDF、活字はOCR |
| **卒業証明書** | 多言語・スタンプ・印影 | B: PDF直接 or C | 言語判定に視覚が有効 |
| **成績証明書** | 表形式、科目名 | C: テキスト+PDF | 表構造の正確な読取に両方有効 |
| **雇用条件通知書** | テキスト中心、定型 | A: テキストのみ | OCRで十分、コスト最小 |
| **雇用契約書** | テキスト中心、署名あり | A or B | 署名確認が必要ならB |
| **会社資料（Excel系）** | 構造化データ | A: テキストのみ | テキスト抽出で十分 |

## 2. 大量ページ（100枚超）の分割戦略

### 2.1 Gemini の制限値

| 項目 | 制限 |
|---|---|
| 画像数上限 | 3,600枚/リクエスト |
| PDFページ上限 | 1,000ページ/PDF（50MB以下） |
| インラインBase64 | リクエスト全体で 20MB → **File API 推奨** |
| File API | 個別ファイル 50MB まで |
| コンテキストウィンドウ | 1M tokens（2.5 Pro/Flash, 3 Flash） |
| PDF 100ページ（2.5系） | 100 × 258 = 25,800 tokens → **1Mの2.6%、余裕** |

### 2.2 在留資格申請での実態

- 1申請人あたり平均 27.5ページ（実質入力対象 5-10ページ）
- **100枚超になるケースは稀**（複数申請人バッチの場合のみ）
- 1申請人分は **常に1リクエストで処理可能**

### 2.3 大量ページが発生する場合の戦略

#### ケース1: 1申請人で書類が多い（30-50ページ）

```
→ 1リクエストで全ページ渡す（推奨）
→ PDF直接なら 50ページ × 258 = 12,900 tokens（余裕）
```

**公式ガイダンス**: 分割するより1リクエストで全部渡す方が精度が高い。

#### ケース2: 複数申請人を一括処理（100-300ページ）

```
→ 申請人単位で分割（1申請人 = 1リクエスト）
→ 案件の独立性が高いため、分割のデメリットなし
```

#### ケース3: 同一会社の大量申請（1,000ページ超）

```
→ Batch API を使用（50%コスト割引、100リクエスト/ジョブ）
→ 申請人単位で並列処理
→ 会社共通情報はコンテキストキャッシュで共有
```

### 2.4 分割戦略まとめ

| シナリオ | ページ数 | 戦略 |
|---|---|---|
| 通常（1申請人） | 5-30 | 1リクエスト、分割不要 |
| 書類多め（1申請人） | 30-100 | 1リクエスト、分割不要 |
| 複数申請人バッチ | 100-500 | **申請人単位で分割** |
| 大量バッチ | 500+ | **Batch API + 申請人単位分割** |

## 3. 在留資格申請に最適な推奨パターン

### 3.1 推奨アーキテクチャ: 段階的アプローチ

```
┌─────────────────────────────────────────────────┐
│           Phase 1: MVP（今すぐ開始）               │
│                                                   │
│  [PDF書類] ──→ [Gemini 2.5 Flash] ──→ [抽出JSON] │
│               PDF直接入力（パターンB）              │
│               258 tokens/ページ                    │
│               月100件 = $1.34                      │
└─────────────┬───────────────────────────────────┘
              │ 精度不足の書類タイプを特定
              ▼
┌─────────────────────────────────────────────────┐
│        Phase 2: ハイブリッド（精度改善）             │
│                                                   │
│  [PDF書類] ──→ [Cloud Vision OCR] ──→ [テキスト]  │
│      │                                    │       │
│      └──→ [Gemini 2.5 Flash] ←────────────┘       │
│           テキスト+PDF併用（パターンC）              │
│           ※精度が必要な書類タイプのみ                │
│           月100件 = $2.69                          │
└─────────────┬───────────────────────────────────┘
              │ 判断が難しい案件を検出
              ▼
┌─────────────────────────────────────────────────┐
│        Phase 3: 高精度判定（必要時のみ）             │
│                                                   │
│  [要確認案件] ──→ [Gemini 2.5 Pro] ──→ [詳細判定] │
│                  技人国チェック・整合性検証           │
│                  ※全案件の10-20%程度                │
│                  月20件（Pro） = $1.07              │
└─────────────────────────────────────────────────┘
```

### 3.2 具体的な処理フロー

```
1. 案件取り込み
   └→ PDF/画像ファイルをCloud Storageに保存

2. 書類分類（Gemini 2.5 Flash + PDF直接）
   └→ パスポート/在留カード/履歴書/... を自動分類
   └→ 書類タイプに応じて処理パターンを選択

3. 項目抽出
   ├→ テキスト系書類（雇用条件通知書等）: パターンA（テキストのみ）
   ├→ 画像系書類（パスポート/在留カード）: パターンB（PDF直接）
   └→ 複合系書類（履歴書/成績証明書）: パターンC（テキスト+PDF）

4. 整合性チェック + 技人国判定
   ├→ 標準案件: Gemini 2.5 Flash
   └→ 要確認案件: Gemini 2.5 Pro にエスカレーション

5. 出力: case_data JSON + レビュー指示
```

### 3.3 モデル選定ガイド

| 用途 | MVP（今） | 将来（3系GA後） | 理由 |
|---|---|---|---|
| **書類分類** | 2.5 Flash | 3.1 Flash-Lite | 最速・最安、分類は単純タスク |
| **項目抽出（通常）** | 2.5 Flash | 3.1 Flash-Lite | コスパ最良、ネイティブPDF対応 |
| **項目抽出（高精度）** | 2.5 Pro | 3 Flash | Pro級性能を Flash 価格で |
| **技人国チェック** | 2.5 Pro | 3 Flash | 推論力が求められる判定 |
| **不足確認・依頼文生成** | 2.5 Flash | 3.1 Flash-Lite | テキスト生成は Lite で十分 |

### 4.5 Gemini 3系の `media_resolution` 活用

3系モデルでは PDF/画像の解像度をリクエスト単位で制御可能:

| 書類タイプ | 推奨設定 | トークン/ページ | 理由 |
|---|---|---|---|
| パスポート（MRZ読取） | HIGH | 1,120 | 細かい文字の正確な読取 |
| 在留カード | HIGH | 1,120 | 小さい文字・写真 |
| 雇用条件通知書 | MEDIUM | 560 | テキスト中心、MEDIUMで十分 |
| 履歴書 | MEDIUM | 560 | 公式推奨「品質はMEDIUMで飽和」 |
| 成績証明書（表形式） | HIGH | 1,120 | 表構造の正確な読取 |

> 公式ガイダンス: 「品質は MEDIUM で飽和する傾向がある」 → デフォルトを MEDIUM にし、精度が必要な書類のみ HIGH に上げるのが最適。

## 5. コスト最適化のヒント

### 5.1 即効性のある施策

| 施策 | 効果 | 実装難易度 |
|---|---|---|
| **PDF直接入力**（画像変換を避ける） | トークン1/10 | 低 |
| **Batch API** | 50%コスト割引 | 低 |
| **コンテキストキャッシュ**（同一会社の複数申請） | 最大75%削減 | 中 |
| **書類タイプ別の処理パターン切替** | 不要なOCR削減 | 中 |
| **実質入力対象ページのみ処理**（27.5→7.5ページ） | トークン73%削減 | 中 |

### 5.2 コスト構造の理解

- **出力コストが支配的**: 入力方式の違いより、出力 5,000 tokens の費用が大きい
- **Vision API OCR 費用**: $0.011/申請人で無視できるレベル
- **全パターンで月$30以下**: コストは意思決定の主要因にならない → **精度優先で選択すべき**

### 5.3 月間コスト早見表（推奨構成: 2.5 Flash + PDF直接）

| 月間件数 | Gemini API | Vision OCR | 合計 |
|---|---|---|---|
| 50件 | $0.67 | $0.57 | **~$1.24** |
| 100件 | $1.34 | $1.13 | **~$2.47** |
| 300件 | $4.02 | $3.39 | **~$7.41** |

※ Batch API 利用で Gemini API 分はさらに半額
