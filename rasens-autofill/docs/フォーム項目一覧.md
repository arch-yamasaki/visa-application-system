### 重要な発見

#### 1. employment は2層構造

| 層 | 項目例 | フォーム上の欄 | データソース |
|---|---|---|---|
| **フォーム入力層** | 月額給与、就労予定期間、職種、活動内容 | あり | フォーム直接入力 |
| **書類レビュー層** | 年収、賞与、労働時間、休日、保険、手当 | **なし** | 雇用条件通知書（添付書類） |

現行 schema.py の `employment_conditions` の多くの項目（annual_salary, bonus, working_hours, holidays, insurance, allowances）はフォームに入力欄がなく、添付書類からの抽出・レビュー用。canonical v2 では、フォーム投入対象もレビュー対象も `employment.*` に寄せ、フォーム投入するかどうかは `review_field_catalog.md` と `application_data.rows` で分ける。

#### 2. フォームにあるが schema.py にない必須項目

全件入力されているが visa-app で未抽出の項目：

| 項目 | 重要度 | 備考 |
|---|---|---|
| 入国予定年月日 | 高 | 書類からは抽出困難（事務所判断） |
| 滞在予定期間 | 高 | 同上 |
| 実務経験月数 | 高 | 職歴から算出可能 |
| 職種（RASENSのプルダウン） | 高 | `employment.activity_details` や雇用条件通知書から変換が必要 |
| 支店・事業所名 | 中 | 会社書類から抽出可能 |
| 雇用保険適用事業所番号 | 中 | 別途資料 |
| 外国人職員数 / 技能実習生数 | 中 | 会社書類から抽出困難（手入力が多い） |

#### 3. 職歴の実態

| パターン | 件数 | 典型 |
|---|---|---|
| 0社（学歴ルート） | 1/5 | 新卒採用 |
| 1社 | 3/5 | 最も多い。母国での1社 |
| 4社 | 1/5 | 職歴豊富（修士+9年経験） |

MVP では 3社まで対応で 4/5件（80%）をカバー。

#### 4. 在日親族

全5件とも0名。フォースバレー経由の新規入国（COE）案件では在日親族がいないケースが大半。MVP では 2名まで対応で十分。

#### 5. システム必須 vs 実データ必須の対照表

3つの情報源を突合した結果。
- **システム必須**: rasens_offer_fields.json の `required: true`（HTML の `required_span` と完全一致、121箇所）
- **実データ必須**: 5件の実データで全件に値が入っている項目
- **条件付き必須**: `conditional: true` で、親項目の回答次第で必須になる項目

##### 発見: システム必須 ≠ 実データ必須

| パターン | 件数 | 例 |
|---|---|---|
| システム必須 かつ 実データ必須 | 約40項目 | 氏名、生年月日、月額給与等 |
| システム任意 だが 実データ必須 | 約8項目 | **国籍**、旅券番号、電話番号、支店名、雇用保険番号、技能実習生数 |
| システム必須 だが 実データ未入力 | 約15項目 | 条件付き必須（出入国歴詳細、在日親族詳細等。親項目が「無」のため） |
| システム任意 かつ 実データ任意 | 約10項目 | 携帯電話番号、他業種、派遣先等 |

**注目**: 国籍（No.1）はシステム上 `required: false` だが、実データでは全件入力。これはシステムの必須設定ミスか、条件付きロジックの影響の可能性がある。
