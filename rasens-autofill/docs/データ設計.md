# データ設計

## 基本方針

在留資格申請のデータは、次の3層に分ける。

1. `case_data`: 案件の正本。実運用では visa-app の Firestore `cases/{case_id}.case_data` に保存する。
2. `form_definitions`: 入管オンライン申請フォームの項目台帳。画面上の `field_id`、`field_name`、選択肢、必須、条件付き項目を保持する。
3. `application_data`: Chrome拡張へ渡す投入用JSON。`case_data`、フォーム台帳、mapping から backend が生成する派生物。

`rasens-autofill/data/cases/` は合成デモ・fixture用であり、実運用案件の正本ではない。`application_data.json` もブラウザ入力用の薄い派生物であり、実務判断、OCR根拠、レビュー状態、添付資料管理を保持する場所ではない。

### visa-app API連携

Chrome拡張は visa-app の `/cases/{case_id}/application-data` エンドポイントから `rows` を取得する。Firestore `case_data` は canonical な値だけの正本にし、backend 側で `case_data + form_definitions + mappings -> application_data rows` を生成する。Chrome拡張は rows を RASENS DOM に入力するだけの責務に寄せる。

この方針は決定済みです。`transform` と `visible_when` は backend generator だけで処理し、Chrome拡張やデモ生成CLIには再実装しない。

テスト用途では、まず `test_cases_from_raw/` の単票ケースに絞る。オンライン申請フォームは1申請人=1フォームなので、1人分の原資料から `case_data` と `application_data` を作れるかを先に検証する。

## Git管理方針

`rasens-autofill/data/` は、アプリと変換ロジックの設計資産として git 管理する。

- `schemas/`: `case_data` の構造契約。
- `cases/demo_case_data.json`: 合成デモ案件。実PIIを入れない。
- `form_definitions/`: 入管オンライン申請フォームの項目台帳。
- `mappings/`: `case_data` からフォーム項目への変換ルール。
- `generated/demo_application_data.json`: デモ案件から再生成できる投入例。
- `reference_form.html`: フォーム台帳を作るための参照元。実PIIを含まない前提。

`visa-eval/` は、Codex やAIエージェントで実PDF・Excelを読み取り、正解データを作れるか検証する作業領域。実案件由来の `raw/`、`catalog.json`、`test_cases_from_raw/<case_id>/` は `restricted test data` としてローカル管理し、git 管理しない。

git に載せてよいのは、`visa-eval/README.md` と `visa-eval/eval_config/suites/*.json` のような、実PIIを含まない運用説明・評価スイートだけにする。

## 正規 case_data

`case_data` は、申請画面の項目順ではなく、案件・実務判断の単位で持つ。

主なトップレベル構造。canonical v2 では top-level を増やしすぎず、申請人に属する情報は `applicant.*` 配下へ寄せる。

- `case`: 案件ID、流入経路、申請種別、対象在留資格、ワークフロー状態。
- `applicant`: 申請人の基本情報、旅券、出入国歴、家族、学歴、職歴、資格。
- `entry_plan`: 主たる活動内容、入国目的、予定日、希望期間、査証申請予定地。
- `employer`: 所属機関・雇用会社。
- `employment`: 今回の契約・就労条件・活動内容詳細。
- `proxy`: 代理人。受入企業側の担当者を案件ごとに確認する。
- `settings.intermediary`: 取次者。太田さん側の申請アカウントを持つ申請会社情報を固定設定値として使う。
- `receiving_method`: 受領方法。

`passport`, `application`, `family`, `immigration_history`, `education`, `employment_history`, `contract`, `activity_details`, `employment_conditions` は旧pathとして扱い、正本では使わない。旅券は `applicant.passport.*`、契約形態と活動内容は `employment.*` に置く。

RASENS上の `主たる活動内容`、`入国目的`、`活動内容詳細` は別項目として扱う。`主たる活動内容` は `entry_plan.main_activity_category`、画像で確認した身分事項 No.11 の `入国目的` は `entry_plan.purpose_of_entry`、所属機関セクション No.11 の600文字textareaは `employment.activity_details` に置く。

添付資料一覧、OCR根拠、再依頼状態、AIまたは人の判断結果は、`case_data` の値本体ではなく visa-app の `document_manifest`, `field_metadata`, `review` 側で扱う。

現行定義は [case_data.schema.json](/Users/yohei/Documents/1.起業/31.%20在留資格申請/rasens-autofill/data/schemas/case_data.schema.json) に置く。これは canonical v2 の基本構造契約であり、レビュー規則やRASENSの条件付き必須までは持たせない。

## フォーム台帳

[rasens_offer_fields.json](/Users/yohei/Documents/1.起業/31.%20在留資格申請/rasens-autofill/data/form_definitions/rasens_offer_fields.json) がフォーム項目台帳の正本。

この台帳は `reference_form.html` から抽出したもので、次を保持する。

- 項目番号
- ラベル
- `field_id`
- `field_name`
- `input_type`
- `select` の選択肢
- 必須項目
- 条件付き項目

17.2〜17.4 のような条件付き項目、21.2〜21.8 のような在日親族の繰り返し項目、職歴01〜06のような繰り返し項目もフォーム台帳側で保持する。

## マッピング

[rasens_offer_mapping_v2.json](/Users/yohei/Documents/1.起業/31.%20在留資格申請/rasens-autofill/data/mappings/rasens_offer_mapping_v2.json) は、`case_data` の安定キーをフォーム項目へ対応させる。`rasens_offer_fields.json` を正として `field_id` / `field_name` を照合する。

例。

```json
{
  "canonical_path": "applicant.immigration_history.latest_entry.start_date",
  "source_paths": ["applicant.immigration_history.latest_entry.start_date"],
  "form_order": 27,
  "display_no": "17.3",
  "field_name": "item[31].textData",
  "field_id": "switch_256127",
  "input_type": "text",
  "transform": "date_yyyymmdd"
}
```

`field_id` や `field_name` は画面依存なので、案件正本のキーには使わない。`display_no` は人間向けの参考番号、`form_order` はフォーム台帳内の並び順であり、どちらも業務IDとして使わない。画面変更時は `form_definitions` と `mappings` を更新し、`case_data` は維持する。

条件付き項目は `visible_when` で表す。例えば17.2〜17.4は、17.1「過去の出入国歴 有無」が有の場合だけ投入対象になる。

繰り返し項目は `groups` で管理する。例えば在日親族21.2〜21.8は最大6枠、職歴26.1〜26.6も最大6枠の物理フォーム項目として存在するが、`case_data` 側では配列として扱う。

現時点のマッピングはMVP自動投入対象に絞った初期版。フォーム全項目の正は `form_definitions/rasens_offer_fields.json` にあり、全274行台帳からMVP対象と手入力・設定値・非対応を拡張していく。

274行台帳では、自動投入しない行にも扱いを付ける。`manual`, `settings`, `derived`, `unsupported`, `future` のように分類し、未対応なのか、固定設定値で入れるのか、将来対応なのかを明確にする。自動投入はMVP対象だけに絞る。

`proxy` は案件データとして扱い、住所・電話は `employer.*` から初期化候補を作る。`proxy.name` は会社名ではなく受入企業側の担当者名なので、人確認または手入力が必要。

`intermediary` は案件書類から抽出しない。太田さん側の申請アカウントを持つ申請会社情報を backend の設定値として rows 生成時に注入する。

## 型・enum・required の扱い

`reference_form.html` / `rasens_offer_fields.json` から、`input_type`、画面上の必須、select/radio の選択肢、`max_length`、条件付き項目はかなり推測できる。ただし、これは RASENS入力制約であり、Firestore `case_data` の業務スキーマそのものではない。

使い分けは次の通り。

- `case_data.schema.json`: 申請データの意味、canonical path、基本型を定義する。
- `form_definitions`: RASENSの物理項目、選択肢、画面上の必須、文字数制限を保持する。
- `review/completeness rules`: 実務上の不足、条件付き必須、手入力・固定値・非対応を判定する。

例えば、取次者はRASENS画面上は必須でも、Gemini抽出の required ではなく固定設定値です。電話番号と携帯電話番号のような「どちらか必須」も、単純な JSON Schema の `required` だけでは表せないため、レビュー規則で扱う。

### 現時点の投入カバレッジ

フォーム台帳は274項目規模だが、現時点の `rasens_offer_mapping_v2.json` はMVP主要項目に絞った初期版。Chrome拡張の `application_data` としては、申請人基本情報、旅券、入国予定、出入国歴の一部、学歴の一部、所属機関の一部、活動内容詳細を生成できる。

一方で、Excel intake だけでは次が不足しやすい。

- 性別、日本連絡先、退去強制・出国命令歴など、Excelにない申請人項目。
- 代理人・取次者、受領方法、確認チェック、顔写真など、運用側で決める項目。
- 学歴区分、専攻・専門分野、成績科目、職歴2件目以降、在日親族の詳細。
- 雇用条件通知書や会社書類から拾う契約条件、会社規模、活動内容詳細。

したがって、`application_data.golden.json` は「MVPマッピングから生成できる部分正解」として扱う。フォーム送信可能な完全データにするには、PDF・会社書類・人手確認で `case_data.golden.json` を補完し、マッピングを追加する。

## デモJSON

デモ用には2種類のJSONを置く。

- [demo_case_data.json](/Users/yohei/Documents/1.起業/31.%20在留資格申請/rasens-autofill/data/cases/demo_case_data.json): 架空案件の fixture。
- [demo_application_data.json](/Users/yohei/Documents/1.起業/31.%20在留資格申請/rasens-autofill/data/generated/demo_application_data.json): Chrome拡張に渡す投入用JSON。

`demo_application_data.json` は手編集せず、次のコマンドで生成する。

```bash
python3 rasens-autofill/scripts/build_application_data.py \
  rasens-autofill/data/cases/demo_case_data.json \
  rasens-autofill/data/mappings/rasens_offer_mapping_v2.json \
  rasens-autofill/data/generated/demo_application_data.json
```

## 運用上の注意

- 実案件データとデモデータを混ぜない。
- `visa-eval/test_cases_from_raw/` の単票ケースを、Codex評価とフォーム投入確認の入口にする。
- Excel intake だけではフォーム必須項目を満たせないことがあるため、PDF束、会社書類、雇用条件通知書、人手確認で `case_data` を補完する。
- 個人情報を含む実案件の `case_data` は、共有範囲と保存場所を明確にする。
- 公開・共有可能なfixtureは、合成データまたは匿名化済みデータに限定する。
- 実案件raw資料は制限付きデータとして扱い、必要に応じてリポジトリ外の保管領域へ分離する。
- `unknown`、空欄、`not_applicable` は区別する。未回収と非該当が混ざると不足確認が壊れる。
- `confidence=medium` やOCR根拠が弱い項目は人確認に回す。
- Chrome拡張の `application_data.json` は同梱デモ・検証用。案件正本ではない。
- 最終送信は自動化しない。入力後は人が確認してから手動で進める。
