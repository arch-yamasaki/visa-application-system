# フォルダ整理案

## 目的

在留資格申請AI評価では、次の3種類のデータが混ざるとすぐ分かりにくくなります。

- 原資料: PDF、Excel、Docx、メールなど。
- 正解データ: `expected/*.golden.json`。
- AI実行結果: `generated/*.json`。

特に、AIにPDF等を読ませる検証では、AIが `expected/*.golden.json` を見てしまうと評価不能になります。そのため、フォルダ構成は「保管場所」と「AI実行場所」を明確に分けます。

## 現在の基本方針

現在は次の分け方にします。

```text
visa-eval/
  raw/                 # 原資料保管。AIに直接渡す入口ではない
  catalog.json          # rawの分類結果。restricted
  test_cases_from_raw/      # 単票fixture。expected/goldenを含む保管場所
  blind_runs_from_test_cases/                 # AIに渡すブラインド実行場所
  eval/                 # 共有可能な評価定義・プロンプト
```

最重要の整理はこれです。

```text
test_cases_from_raw/ は「保管・正解比較用」
blind_runs_from_test_cases/ は「AI実行用」
```

## 各フォルダの役割

| フォルダ | 役割 | AIに見せるか |
| --- | --- | --- |
| `raw/` | 原資料の保管場所 | 原則そのまま見せない |
| `test_cases_from_raw/` | 単票fixtureの保管場所。`expected/` を含む | 見せない |
| `test_cases_from_raw/*/*/input/` | 元fixtureの入力manifest | 直接は見せない |
| `test_cases_from_raw/*/*/expected/` | golden正解 | 絶対に見せない |
| `blind_runs_from_test_cases/<run_id>/` | AI実行用の隔離ディレクトリ | 見せる |
| `blind_runs_from_test_cases/<run_id>/documents/` | AIが読む資料へのsymlink | 見せる |
| `blind_runs_from_test_cases/<run_id>/generated/` | AI出力先 | 書かせる |
| `eval_config/prompts/` | AI用プロンプトテンプレート | 見せてもよい |
| `eval_config/suites/` | 評価スイート定義 | 通常は人間・runner用 |

## なぜ `blind_runs_from_test_cases/` を分けるのか

`test_cases_from_raw/` の中には `expected/*.golden.json` があります。

```text
test_cases_from_raw/<case>/<applicant>/
  input/
  expected/   <- AIに見せると評価不能
  generated/
```

このフォルダをAIに丸ごと渡すと、正解を見てから出力できてしまいます。

そのため、`prepare_blind_eval_run.py` で次のようなAI専用フォルダを作ります。

```text
blind_runs_from_test_cases/<run_id>/
  AGENT_TASK.md
  scenario.json
  document_manifest.blind.json
  documents/
  generated/
```

ここには `expected/` も `*.golden.json` も入りません。

## `blind_runs_from_test_cases/<run_id>` の命名

現在の命名は次です。

```text
YYYYMMDD_HHMMSS_<case_id>
```

例:

```text
20260509_231147_gijinkoku_a_company_round1__amit_tamang
```

この形式だと、次が分かります。

- いつ実行したか。
- どのケースを実行したか。
- 同じケースを何回も試せる。

## 今後の拡張案

runが増えてきたら、`blind_runs_from_test_cases/` をさらに状態別に分けると見やすくなります。

```text
visa-eval/
  blind_runs_from_test_cases/
    active/
      20260509_231147_gijinkoku_a_company_round1__amit_tamang/
    reviewed/
      20260510_101500_gijinkoku_a_company_round1__amit_tamang/
    archived/
      20260508_180000_old_test/
```

ただし、最初はここまで分けなくてよいです。まずは `blind_runs_from_test_cases/<run_id>/` だけで十分です。

## さらに分かりやすくする改善案

### 1. `blind_runs_from_test_cases/latest` symlink

直近のrunへすぐ移動できるようにします。

```text
visa-eval/blind_runs_from_test_cases/latest -> 20260509_231147_gijinkoku_a_company_round1__amit_tamang
```

メリット:

- AIに「`blind_runs_from_test_cases/latest/AGENT_TASK.md` を読んで」と言える。
- 毎回長いrun IDをコピーしなくてよい。

注意:

- `latest` はローカル専用。
- git管理しない。

### 2. `run_manifest.json`

各runに実行メタデータを置く案です。

```json
{
  "run_id": "20260509_231147_gijinkoku_a_company_round1__amit_tamang",
  "source_fixture": "visa-eval/test_cases_from_raw/gijinkoku_a_company_round1/amit_tamang",
  "case_id": "gijinkoku_a_company_round1__amit_tamang",
  "mode": "blind_extraction",
  "expected_access": "forbidden",
  "status": "prepared"
}
```

メリット:

- prepared / running / completed / reviewed などの状態管理ができる。
- どのfixtureから作ったrunか追いやすい。

### 3. `comparisons/` を分ける

AI出力とgolden比較の結果は、AI実行フォルダとは分ける案です。

```text
visa-eval/
  comparisons/
    20260509_231147_gijinkoku_a_company_round1__amit_tamang/
      diff_summary.md
      case_data.diff.json
      review.diff.json
      application_data.diff.json
```

理由:

- `blind_runs_from_test_cases/` はAIがgoldenを見ない場所。
- `comparisons/` は人間がgoldenと比較する場所。
- 役割が混ざらない。

この案は、比較runnerを作る段階で導入すればよいです。

## git管理方針

git管理する:

- `visa-eval/README.md`
- `visa-eval/test_cases_from_raw/README.md`
- `visa-eval/eval_config/suites/*.json`
- `visa-eval/eval_config/prompts/*.md`
- `docs/003_data_and_testing/*.md`

git管理しない:

- `visa-eval/raw/`
- `visa-eval/catalog.json`
- `visa-eval/test_cases_from_raw/<case_id>/`
- `visa-eval/blind_runs_from_test_cases/`
- `visa-eval/comparisons/`
- `generated/`
- 実案件由来のChrome投入JSON

## 結論

当面はこの構成で進めるのがよいです。

```text
test_cases_from_raw/ = 正解付きの保管場所
blind_runs_from_test_cases/            = AIに渡すブラインド実行場所
eval/            = 共有可能な評価定義・プロンプト
```

次に実装するなら、優先度は次です。

1. `blind_runs_from_test_cases/latest` symlink を `prepare_blind_eval_run.py` で作る。
2. `run_manifest.json` を各runに追加する。
3. golden比較結果を `comparisons/` に出す比較runnerを作る。
