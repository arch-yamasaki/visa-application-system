# 単票評価ワークフローまとめ

## このドキュメントの目的

このドキュメントは、在留資格申請AIエージェントのテストデータ設計を、最初に読む人向けにまとめた入口です。

今の方針はシンプルです。

```text
まずは「申請人1人分」をちゃんとできるか検証する。
```

オンライン申請フォームは基本的に `1申請人 = 1フォーム` です。そのため、複数人をまとめたバッチケースではなく、`test_cases_from_raw/` の単票ケースを中心にします。

## 全体像

登場するデータは、大きく4種類です。

```text
raw資料
  -> document_manifest
  -> case_data
  -> application_data
  -> Chrome拡張でフォーム入力
```

それぞれの意味は次の通りです。

| 名前 | 役割 | 例 |
| --- | --- | --- |
| `raw/` | 受領した原本資料。PDF、Excel、Docx、メールなど。加工しない。 | 旅券PDF、申請人資料束、雇用条件通知書、会社書類 |
| `document_manifest.json` | AIエージェントに渡す入力資料リスト。rawへのパス、文書種別、使うかどうかを持つ。 | submitted_application_pdf, employment_terms |
| `case_data.golden.json` | 案件の正本候補。申請人、旅券、学歴、職歴、会社、レビュー根拠を正規化して持つ。現状はschemaと一部ズレがある。 | `applicant.name_roman`, `passport.number` |
| `application_data.golden.json` | Chrome拡張の入力契約と同じ形式のフォーム投入用JSON。`case_data` から生成される派生物。 | `field_id`, `field_name`, `fill_value` |
| `review.golden.json` | 不足項目、リスク、人手確認の正解。 | 性別不足、退去強制歴未確認など |

重要なのは、`case_data` が正本で、`application_data` は派生物という点です。

## ディレクトリ構成

実案件由来の評価データは、リポジトリ直下の `visa-eval/` に置きます。

```text
visa-eval/
  raw/
    申請書類/
  catalog.json
  test_cases_from_raw/
    <case_id>/
      <applicant_id>/
        scenario.json
        input/
          document_manifest.json
        expected/
          case_data.golden.json
          application_data.golden.json
          review.golden.json
        generated/
  eval_config/
    suites/
      single_smoke.json
```

例:

```text
visa-eval/
  test_cases_from_raw/
    gijinkoku_a_company_round1/
      amit_tamang/
        scenario.json
        input/document_manifest.json
        expected/case_data.golden.json
        expected/application_data.golden.json
        expected/review.golden.json
        generated/
```

`fixtures/` は使いません。まず単票だけに絞ります。

## 1ケースの中身

### `scenario.json`

その単票ケースのメタ情報です。

```json
{
  "case_id": "gijinkoku_a_company_round1__example_applicant",
  "base_case_id": "gijinkoku_a_company_round1",
  "applicant_id": "example_applicant",
  "target_status": "engineer_humanities_international",
  "application_type": "certificate_of_eligibility",
  "primary_checks": [
    "document_completeness",
    "case_data_generation",
    "form_mapping",
    "gijinkoku_fit"
  ]
}
```

`case_id` は `元ケースID__申請人ID` の形にします。

### `input/document_manifest.json`

AIエージェントに読ませる資料リストです。

```json
{
  "case_id": "gijinkoku_a_company_round1__example_applicant",
  "base_case_id": "gijinkoku_a_company_round1",
  "applicant_id": "example_applicant",
  "documents": [
    {
      "document_role": "submitted_application_pdf",
      "path": "visa-eval/raw/申請書類/.../EXAMPLE.pdf",
      "file_name": "EXAMPLE.pdf",
      "use_as_input": false
    },
    {
      "document_role": "employment_terms",
      "path": "visa-eval/raw/申請書類/.../offer_letter.pdf",
      "file_name": "offer_letter.pdf",
      "use_as_input": true
    },
    {
      "document_role": "intake_spreadsheet",
      "path": "visa-eval/raw/申請書類/.../COE書類.xlsx",
      "file_name": "COE書類.xlsx",
      "use_as_input": true
    }
  ]
}
```

主な `document_role` は次です。

| role | 意味 | 期待する使い方 |
| --- | --- | --- |
| `submitted_application_pdf` | 提出済み申請書PDF | golden正解データの根拠。AIインプットには使わない |
| `employment_terms` | 雇用条件通知書、オファーレター | 職務、給与、契約期間、勤務地を読む |
| `company_documents` | 会社書類 | 会社名、住所、資本金、売上、従業員数などを読む |
| `intake_spreadsheet` | 紹介会社等のExcelヒアリング表 | 基本情報の一次ソースとして使う |
| `unused_resume` | 履歴書（申請には添付しない） | 申請添付はしないが、職歴・学歴等はAI抽出の対象 |
| `not_attached_reference` | 添付外の参考資料 | 差分確認や補助情報に使う |
| `unused_resume` | 申請には添付しない履歴書 | 申請添付はしないが、記載情報（職歴・学歴等）はAI抽出の対象 |

## `expected/` の3ファイル

### 1. `case_data.golden.json`

一番重要な正解データです。案件の正本候補です。

例:

```json
{
  "case": {
    "case_id": "gijinkoku_a_company_round1__example_applicant",
    "target_status": "engineer_humanities_international",
    "workflow_state": "golden_partial"
  },
  "applicant": {
    "name_roman": "EXAMPLE APPLICANT",
    "birth_date": "1990-01-01",
    "nationality_region": "ネパール Nepal",
    "sex": "unknown"
  },
  "passport": {
    "number": "MASKED",
    "expiry_date": "2030-01-01"
  },
  "education": [
    {
      "school_name": "Example University",
      "graduation_date": "2020-03-31",
      "major": "Computer Science"
    }
  ],
  "employer": {
    "name": "Example Company"
  }
}
```

このファイルは、AIが最終的に作るべき正規化データです。フォームの `field_id` や `field_name` はここには持たせません。画面が変わっても `case_data` は維持したいからです。

注意点として、現時点の restricted fixture は、厳密には `rasens-autofill/data/schemas/case_data.schema.json` に完全準拠していません。例えば `workflow_state` に `golden_partial` を使う、未確認値に `null` を使う、といった運用上の値が残っています。したがって、今の `case_data.golden.json` は「schema完全準拠の完成正解」ではなく、「人手で完成させていく正本候補」として扱います。

### 2. `application_data.golden.json`

Chrome拡張の入力契約と同じ形式のフォーム投入用JSONです。`case_data` と `rasens_offer_mapping.json` から生成します。

例:

```json
[
  {
    "label": "主たる活動内容",
    "canonical_id": "application.desired_status_label",
    "field_id": "select_256097",
    "field_name": "item[1].selectData",
    "input_type": "select",
    "fill_value": "技術・人文知識・国際業務"
  },
  {
    "label": "氏名",
    "canonical_id": "applicant.name_roman",
    "field_id": "applicant_lastName",
    "field_name": "item[7].textData",
    "input_type": "text",
    "fill_value": "EXAMPLE APPLICANT"
  }
]
```

これは正本ではありません。フォーム入力のための派生データです。

また、`application_data` はフォーム全項目の完全表ではありません。`fill_value` が空、`unknown`、`not_applicable`、または条件付き項目が非表示の場合、その行は出力されません。つまり、行が存在しない理由は「未マッピング」とは限らず、「値が未回収」「非該当」「表示条件を満たしていない」場合もあります。

`application_data.golden.json` は、次のコマンドで再生成できます。

```bash
python3 rasens-autofill/scripts/build_application_data.py \
  visa-eval/test_cases_from_raw/<case_id>/<applicant_id>/expected/case_data.golden.json \
  rasens-autofill/data/mappings/rasens_offer_mapping.json \
  visa-eval/test_cases_from_raw/<case_id>/<applicant_id>/expected/application_data.golden.json
```

### 3. `review.golden.json`

不足確認、リスク判定、人手確認の期待値です。

例:

```json
{
  "expected_route": "needs_review",
  "missing_items": [
    {
      "path": "applicant.sex",
      "reason": "intake_spreadsheet_does_not_include_sex"
    }
  ],
  "findings": [
    {
      "code": "pdf_bundle_not_fully_verified",
      "severity": "medium",
      "message": "PDF and company documents still need verification."
    }
  ]
}
```

現状の `review.golden.json` には、Excel起点で自動生成したscaffoldが含まれます。最終的なレビュー正解にするには、PDF、雇用条件通知書、会社書類を見て人手で更新する必要があります。

## データ生成の流れ

### 1. raw資料を分類する

`raw/申請書類` を更新したら、まず catalog を作ります。

```bash
python3 rasens-autofill/scripts/classify_test_documents.py
```

これは `visa-eval/catalog.json` を更新します。`test_cases_from_raw/` は直接生成しません。

### 2. Excel intakeから単票fixtureのたたき台を作る

```bash
python3 rasens-autofill/scripts/build_golden_from_intake.py
```

このスクリプトは、Excel intakeを一次ソースとして `test_cases_from_raw/` の `expected/*.golden.json` を作ります。

ただし、ここでできるのは「たたき台」です。Excelにない項目は不足します。

### 3. 人手でgoldenを補完する

PDF、雇用条件通知書、会社書類、申請書類束を見て、`case_data.golden.json` を補完します。

補完が必要になりやすい項目:

- `applicant.sex`
- `applicant.japan_contact.*`
- `immigration_history.deportation_or_departure_order`
- `application.activity_details`
- `employment_history`
- `employer.*`
- `proxy`
- `intermediary`
- `receiving_method`
- `family.japan_relatives_or_cohabitants`
- `transcript_subjects`

### 4. `application_data` を再生成する

`case_data.golden.json` を直したら、必ず `application_data.golden.json` を再生成します。

```bash
python3 rasens-autofill/scripts/build_application_data.py \
  <case_data.golden.json> \
  rasens-autofill/data/mappings/rasens_offer_mapping.json \
  <application_data.golden.json>
```

## Chrome拡張との関係

Chrome拡張は `case_data` を直接読みません。読むのは `application_data` の行配列です。

```text
case_data.golden.json
  + rasens_offer_mapping.json
  -> application_data.golden.json
  -> extension/application_data.json など拡張が読める場所へ反映
  -> Chrome extension
  -> 入管オンライン申請フォーム
```

現状の拡張UIは、fixture配下の `application_data.golden.json` を直接選んで読み込む実装ではありません。拡張の「同梱データを使う」は `rasens-autofill/extension/application_data.json` を読み、`chrome.storage.local` に保存します。

そのため、fixtureの実データでChrome確認する場合は、手順を分けます。

```text
test_cases_from_raw/.../expected/application_data.golden.json
  -> 検証用に extension/application_data.json へ反映する
  -> 拡張をリロードする
  -> 「同梱データを使う」
```

実案件データを `extension/application_data.json` に入れたままgit管理・共有しないでください。

Chrome拡張が1行ごとに見る主なキー:

| キー | 意味 |
| --- | --- |
| `label` | 表示・ログ用のラベル |
| `field_id` | DOMのid。あれば最優先で探す |
| `field_name` | DOMのname。radio/selectでは重要 |
| `input_type` | `text`, `select`, `radio`, `textarea` など |
| `fill_value` | 実際に入力する値 |
| `canonical_id` | 元の `case_data` キー |

`no` はフォーム上の項目番号ですが、空欄や重複があります。一意キーとして扱わず、ログ・確認用の補助情報と考えます。拡張側は主に `field_id`、次に `field_name`、最後に `label` による探索で対象要素を探します。

例:

```json
{
  "label": "入国目的",
  "canonical_id": "application.purpose_of_entry",
  "field_id": "select_256117",
  "field_name": "item[21].selectData",
  "input_type": "select",
  "fill_value": "技術・人文知識・国際業務"
}
```

`field_id` や `field_name` はフォーム画面に依存します。入管サイト側のHTMLが変わったら、`form_definitions/` と `mappings/` を更新します。

## Excelだけでは足りない理由

現状のExcel intakeは便利ですが、フォーム全項目を埋めるには不足します。

例:

| 項目 | Excelだけで足りるか | 補完元 |
| --- | --- | --- |
| 氏名、生年月日、旅券番号 | 足りることが多い | Excel、旅券PDF |
| 性別 | 足りないことがある | 旅券PDF、申請書類PDF |
| 日本連絡先 | 足りないことが多い | 会社・本人確認 |
| 雇用条件、給与、勤務地 | 足りないことが多い | 雇用条件通知書、オファーレター |
| 会社の資本金、売上、従業員数 | 足りないことが多い | 会社書類 |
| 活動内容詳細 | 足りない | 雇用条件、職務内容、人手作成 |
| 技人国の適合判断 | 足りない | 学歴、成績、職務内容、人手確認 |
| 代理人・取次者 | Excelとは別管理になりやすい | 運用側固定値 |
| 受領方法 | Excelとは別管理になりやすい | 運用側固定値 |

そのため、現在の `application_data.golden.json` は「現行マッピングで生成できる範囲の部分正解」として扱います。フォーム送信可能な完全正解ではありません。

## 現時点のマッピング範囲

`rasens-autofill/data/mappings/rasens_offer_mapping.json` は、主要項目だけの初期版です。

今できること:

- 主たる活動内容
- 国籍・地域
- 生年月日
- 氏名
- 出生地
- 配偶者の有無
- 職業
- 本国住所
- 旅券番号・有効期限
- 入国目的
- 入国予定日
- 上陸予定港
- 滞在予定期間
- 出入国歴の一部
- COE申請歴の一部
- 犯罪歴
- 在日親族有無
- 最終学歴の学校名・卒業年月日
- 職歴1件目の一部
- 所属機関情報の一部
- 活動内容詳細

まだ弱いところ:

- 在日親族の詳細 `21.2` 以降
- 学歴区分、専攻 `23.1` から `24.*`
- 職歴2件目以降
- 代理人 `27.*`
- 取次者
- 受領方法
- 確認チェック
- 顔写真、添付ファイル
- 高度専門職ポイント表

## 現時点の制約

現状を過大評価しないため、次の前提を明確に置きます。

- 評価対象は単票のみ。複数申請人をまとめたバッチ評価は今は扱わない。
- `case_data.golden.json` は正本候補だが、現行schemaとは一部ズレがある。
- `case_data.golden.json` と `review.golden.json` はExcel intake起点のscaffoldを含む。
- PDF、DOCX、MSG、会社書類は主に補完・検証・差分確認用で、項目別provenanceはまだ整備していない。
- `application_data.golden.json` は疎な行配列。フォーム全項目の完全表ではない。
- Chrome拡張はfixtureファイルを直接読むのではなく、同梱JSONを `chrome.storage.local` に読み込む。
- `review.golden.json` は最終審査結果ではない。人手確認すべき不足・リスクの期待値として育てる。
- `confidence` は現状 `demo` / `generated` が中心で、OCR不確実性を十分に表現できていない。

## 評価モードとGolden作成モード

### Golden作成モード

まだ正解が固まっていないときの流れです。

```text
document_manifest.json
  -> AIが generated/case_data.json を作る
  -> AIが generated/review.json を作る
  -> スクリプトで generated/application_data.json を作る
  -> 人が確認・修正する
  -> expected/*.golden.json に昇格する
```

### 評価モード

正解が固まった後の流れです。

```text
document_manifest.json
  -> AIが generated/*.json を作る
  -> expected/*.golden.json と比較する
```

比較では、自然文の完全一致よりも、次を重視します。

- 必須キーが取れているか
- 氏名、旅券、生年月日などの安定項目が一致しているか
- 不足項目が正しく出ているか
- `application_data` の `field_id` / `field_name` / `fill_value` が正しいか
- 技人国として人手確認すべきリスクが出ているか

## 実データ管理の注意

`visa-eval/` は実PIIを含む restricted workspace です。

git管理するもの:

- `visa-eval/README.md`
- `visa-eval/test_cases_from_raw/README.md`
- `visa-eval/eval_config/suites/*.json`

git管理しないもの:

- `visa-eval/raw/`
- `visa-eval/catalog.json`
- `visa-eval/test_cases_from_raw/<case_id>/`
- `generated/`
- スクリーンショット
- DevTools export
- 実案件由来のChrome投入JSON

Chrome拡張で実データを使うときは、専用Chromeプロファイルを使い、検証後に `chrome.storage.local` を削除します。

## まずやるべき1件の検証

最初は1人だけ選びます。

例:

```text
visa-eval/test_cases_from_raw/gijinkoku_a_company_round1/amit_tamang/
```

確認すること:

1. `input/document_manifest.json` に必要なPDF、Excel、会社書類が入っているか。
2. Codexが `generated/case_data.json` を作れるか。
3. `expected/case_data.golden.json` と差分を見て、どこがExcelだけでは足りないか確認する。
4. `case_data.golden.json` を人手で補完する。
5. `application_data.golden.json` を再生成する。
6. Chrome拡張でプレビューし、フォーム項目が見つかるか確認する。
7. 実値を外部共有せず、必要ならマスクしてバグ報告する。

この1件が通ったら、同じ型で他の申請人へ広げます。
