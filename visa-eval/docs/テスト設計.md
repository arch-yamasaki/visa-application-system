# テスト設計

## 目的

在留資格申請AIエージェントが、原資料から正規 `case_data` を作り、そこからレビュー結果を生成できるかを検証する。

フォーム投入用 `application_data` は、`case_data` とフォーム台帳・マッピングから deterministic に生成する。ここはモデル推論ではなく、変換ロジックの正しさを確認する。

最終的には、資格ごと・申請種別ごとに「入力資料」と「正解アウトプット」を持ち、AIエージェントの出力と比較する。

## テストデータの配置

テストデータは [rasens-autofill/data/test_cases](/Users/yohei/Documents/1.起業/31.%20在留資格申請/rasens-autofill/data/test_cases) に置く。

現時点の `raw/申請書類` は実案件由来の実PIIを含む `restricted test data`。匿名化済みfixtureではない。MVP検証のため実データ利用は許可するが、通常の開発用fixtureと同じ扱いはしない。

```text
test_cases/
  raw/
  catalog.json
  fixtures/
    <case_id>/
      scenario.json
      input/input_documents.json
      expected/case_data.golden.json
      expected/application_data.golden.json
      expected/review.golden.json
      generated/
  fixtures_single/
    <case_id>/<applicant_id>/
      scenario.json
      input/input_documents.json
      expected/case_data.golden.json
      expected/application_data.golden.json
      expected/review.golden.json
      generated/
  eval/suites/
```

`raw/` は受領した原資料の保管場所。`fixtures/` は複数申請人を含むバッチケースの抽出・正規化テスト用。`fixtures_single/` は申請人1人=1フォームの単票生成テスト用。
単票の即時確認には `eval/suites/single_smoke.json` を使う。`fixtures/` のバッチケースは抽出・正規化が主目的なので、フォーム投入検証は `fixtures_single/` 側で行う。

## restricted test data の扱い

実PIIを含む raw、そこから生成した `generated/`、レビュー用スクリーンショット、DevTools出力は、すべて `restricted test data` として扱う。利用目的は抽出精度確認、整合性確認、フォーム投入確認に限定し、外部共有や恒久保存を前提にしない。

| リスク | 何が起きるか | 対策 |
| --- | --- | --- |
| ローカルファイル残存 | `Downloads`、デスクトップ、作業メモ配下に raw PDF、ZIP、書き出しJSON、スクリーンショットが残り、後日バックアップや端末共有時に第三者が閲覧できる。 | 保管先は `rasens-autofill/data/test_cases` 配下など限定ディレクトリに固定する。検証終了時に不要なコピー、ダウンロード物、スクリーンショットを削除する。作業メモや一時フォルダへ複製しない。 |
| `chrome.storage.local` 残存 | Chrome拡張に投入した氏名、住所、旅券番号、雇用情報がブラウザプロファイルに残り、次回QAや別案件で誤表示・誤投入される。 | 実案件QAは専用Chromeプロファイルで実施する。ケース切替時と検証終了時に `chrome.storage.local` を消去する。常用プロファイルでは実PIIテストをしない。 |
| DevTools console / screenshot / generated JSON 残存 | Consoleログ、画面キャプチャ、`generated/application_data.json`、`generated/review.json` に実PIIや自由記述が残り、後で共有・再利用された際に漏えいする。 | DevToolsやスクリプトの出力は最小限にし、実値ログを残さない。生成JSONとスクリーンショットは raw と同じ `restricted` 扱いにする。検証後は不要ファイルを削除し、共有が必要な場合はマスク版を別途作る。 |
| Git / 共有フォルダ混入 | 実PII入りfixtureや生成物を誤ってコミット、push、共有ドライブ同期すると、履歴や複製先から回収困難になる。 | 実PII入りファイルはGit管理・共有同期の対象に入れない。コミット前に `git status` で確認する。公開用fixtureは合成データまたは匿名化済みデータだけに限定する。 |
| bug報告 / チャット転記漏えい | 不具合再現のためにConsole、JSON、画面文言をそのまま貼り、氏名、住所、電話、メール、旅券番号、申請番号、自由記述が会話ログやチケットに残る。 | bug報告やチャットでは必ずマスクして転記する。必要なのは値そのものではなく項目種別、発生条件、差分内容と考える。再現共有はダミー値か匿名化抜粋を使う。 |

## 入力データ

入力は `input/input_documents.json` に集約する。ここには、raw資料へのパス、文書種別、ページ数、Excelシート情報、入力に使うかどうかを記録する。

主な文書種別。

- `applicant_document_bundle`: 申請人ごとの資料束。
- `employment_terms`: 雇用条件通知書、オファーレター、雇用契約書。
- `company_documents`: 会社書類、決算、登記、会社概要など。
- `intake_spreadsheet`: 紹介会社・会社から来たExcel/CSV等の構造化資料。
- `email_context`: メール文脈。
- `submitted_application_bundle`: 申請書類としてまとめられたPDF。
- `not_attached_reference`: 申請では添付しなかった参考資料。
- `unused_resume`: 申請には使わない履歴書。原則として入力から除外する。

## 期待アウトプット

各fixtureの `expected/` 配下に置く。

- `case_data.golden.json`: 人手レビュー済みの正規case_data。最重要の正解データ。
- `application_data.golden.json`: Chrome拡張へ渡すフォーム投入用JSONの正解データ。
- `review.golden.json`: 不足確認、整合性チェック、人レビュー要否、リスクコードの正解データ。

`case_data.golden.json` は案件の正本。`application_data.golden.json` は派生物。

## 評価の流れ

1. `input_documents.json` をAIエージェントへ渡す。
2. AIが `generated/case_data.json` を作る。
3. `generated/case_data.json` を deterministic な変換処理に渡して `generated/application_data.json` を作る。
4. AIが `generated/review.json` を作る。
5. `expected/*.golden.json` と比較する。

比較は次の4段階で行う。

- Schema適合: JSONとして壊れていないか、必須キーがあるか。
- 抽出一致: 旅券、氏名、学歴、職歴、会社情報などの安定キーが一致するか。
- 生成一致: `application_data` の項目番号、field_id/name、fill_value が期待通りか。ここは AI ではなく変換処理のテスト。
- レビュー一致: 不足資料、リスク、技人国適合判定、ルート先が期待通りか。

## 現在のケース

- `gijinkoku_a_company_round1`: A社 1回目申請。技人国COE想定。26資料。
- `gijinkoku_a_company_round2_family_japan`: A社 2回目申請 家族在住パターン。技人国COE想定。5資料。

現時点の2ケースは、いずれも複数申請人を含むバッチケース。`case_data.golden.json` には `applicants[]` として複数人分の正解データを持つ。

オンライン申請フォームへの投入テストは、最終的には1申請人=1フォームになるため、次の2段階で評価する。

1. バッチ抽出テスト: Excel、PDF束、会社書類から複数申請人の `case_data.applicants[]` を正しく作れるか。
2. 単票生成テスト: `applicants[]` の各申請人を個別の `case_data` に分割し、1人分の `application_data` を生成できるか。

`application_data.golden.json` は単票生成テストで埋める。`fixtures/` のバッチケースは抽出・正規化が主目的なので、フォーム投入検証は `fixtures_single/` 側で行う。バッチケースのままでは1つの申請フォームに対応しないため、現時点では空配列または未完成扱いにする。

`fixtures_single/` の `review.golden.json` は、バッチケースから分割して作った scaffold を含む。抽出・投入の形を揃えるためのもので、レビュー判定そのものの最終正解としては、元ケースごとに人手確認して更新する前提にする。

## 注意

- 実データ利用は可。ただし `restricted test data` として管理し、raw資料とgenerated出力の共有範囲を制限する。
- リポジトリに残してよい公開用fixtureは、合成データまたは匿名化済みデータだけにする。
- 実案件rawを使う検証では、専用Chromeプロファイルを使い、検証後に `chrome.storage.local`、ダウンロード物、スクリーンショット、DevTools exportを削除する。
- 実案件データを `extension/application_data.json` に同梱しない。
- `unknown`、空欄、`not_applicable` を分ける。
- 長文理由書や活動内容詳細は、全文一致ではなく重要キー・レビューコード・リスク判定で評価する。
- 最終送信や本番申請番号の発行につながる操作はテストしない。
